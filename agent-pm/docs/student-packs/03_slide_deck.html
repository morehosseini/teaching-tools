Building an AI Construction Planning Agent

From a one-line project brief to a professional CPM schedule — how we designed, critiqued, and deployed an AI planning assistant for the Australian construction industry.

📝 Brief → 💬 PIR → ✅ Review → 📊 WBS+CPM → 📑 Excel → 🌐 Live App
01
Genesis

Can AI Do What a Planner Does?

The Problem

Construction planning is slow, expert-dependent, and inconsistent. A 10-storey apartment in Melbourne takes weeks just to produce a Class 5 schedule. Junior planners make systematic errors. Clients get inconsistent outputs.

The Hypothesis

An LLM can interpret a brief, ask the right questions, and scaffold a first-draft schedule — grounded in real production rates — faster and more consistently than a blank spreadsheet.

"The goal is not to replace the planner. It is to give them a defensible, traceable first draft in minutes."
02
Design

The Five-Stage Workflow

1. Brief Interpretation

User types a thin description. Gemini AI reads it, extracts project type, GFA, storeys, location, risk profile, and flags what's missing.

2. PIR Dialogue

Progressive Information Request. The AI asks 3-5 targeted questions at a time — not a 20-field form. Each default is explained with a rationale.

3. Planning Basis

A formal summary document listing every assumption, default, source, and uncertainty. The planner must explicitly approve before proceeding.

4. WBS + Schedule

Library-grounded activity generation. 45–140 activities depending on project type. Three-point durations. CPM critical path calculated deterministically.

5. Excel Export

8-tab professional workbook: Executive Summary, PIR, Basis of Schedule, WBS, Full Schedule, Procurement, Risk Register, Validation.

03
Expert Review · Stage 1

We Asked Other AIs to Critique the Plan

Grok's Critique

  • Construction is not manufacturing — every site has unique constraints
  • LLMs cannot guess durations accurately — ground everything in productivity data
  • Victorian projects need Building Permits, VBA inspections, soil class (AS 2870)
  • The agent must be brutally transparent about uncertainty

Gemini's Critique

  • Use RAG for historical benchmarks, not LLM memory
  • Pre-construction gap: include Design, Approvals, and Procurement phases
  • Human-in-the-loop is essential — planners do not trust black boxes
  • Conversational PIR, not a static form
💡 Result: We scrapped the original 'AI generates schedule' concept and pivoted to 'AI scaffolds a planner's first draft.'
04
Key Design Decision

The Pivot

❌ What We Abandoned

Full automation. AI generates a complete schedule and hands it over. No human review. LLM calculates durations from memory. Single-step output. Black box.

✅ What We Built

Scaffolded planning. AI drafts, human approves. Two mandatory approval gates. Deterministic CPM engine. Production rates from static CSV library. Full audit trail. Every assumption is visible.

Stage 1 Feedback → Pivot to Human-in-the-Loop
Stage 2 Feedback → Add Conversational PIR
Implementation → Streamlit + Gemini + Python Engine
Deployment → Streamlit Cloud + GitHub
05
Technical Design

The Architecture

🤖 LLM Layer (llm_service.py)

Gemini 2.5 Pro handles: brief interpretation, PIR question generation, WBS scaffolding, narrative writing. It NEVER calculates durations or CPM.

⚙️ Engine Layer (core/)

Deterministic Python handles: CPM forward/backward pass, float calculation, calendar arithmetic (VIC public holidays, RDOs), Pydantic validation.

🖥️ UI Layer (pages/)

Streamlit pages handle: step navigation, session state, human approval gates, form rendering, chart display.

Key principle: The LLM is the 'interpreter and writer.' Python is the 'calculator.' They never swap roles.
06
Data Grounding

The Library System

Every duration in the schedule traces back to a static, version-controlled data file — not an LLM prediction.

📊 production_rates.csv

Activity-level rates: m²/day, linear m/day, floors/week. Source: industry benchmarks. Human review required flag when not found.

📋 wbs_templates.json

Pre-built WBS structures for 11 project typologies: detached house, high-rise residential, commercial fitout, warehouse, school extension...

🔗 construction_logic.csv

Trade sequencing rules: waterproofing before tiling, structure before envelope, services rough-in before linings.

⏱️ procurement_leads.csv

Long-lead items: structural steel 8–14 weeks, facade systems 16–24 weeks, lifts 20–30 weeks, switchboards 10–16 weeks.

Bug Fix · Real Example

The GFA Bug Story

The Brief: "User entered: '12-storey apartment tower, each floor 780 m²'"
The LLM Output: Gemini returned GFA = 4,680 m² (wrong — it calculated 6 floors × 780)
The Consequence: Wrong GFA → wrong project scale → wrong activity count → wrong schedule duration
The Fix: We added a deterministic cross-check in _normalise_project_classification(). Python calculates: 12 × 780 = 9,360 m². The LLM result is overridden.
Lesson: Never trust LLM arithmetic. Always cross-check with deterministic code.

Philosophy

Key Design Principles

1. Transparency First

Every assumption is visible. Every default is explained. No black boxes.

2. Human Approval Gates

The planner must approve the Planning Basis before WBS generation.

3. Library-Grounded

Production rates come from CSV files, not LLM memory.

4. Conservative by Default

When uncertain, the AI flags high risk and asks — it does not guess.

5. Deterministic Maths

CPM, float, dates, and GFA calculations are always Python, never LLM.

6. Full Audit Trail

Every edit and override is logged with source and rationale.

Takeaways

Summary

🤝 AI + Human Partnership

The most effective AI tools augment expert judgment. Every decision was made to keep the human planner in control.

🔬 Iterative Design

The final app looks nothing like the first plan. Two rounds of AI critique produced a more robust tool.

⚙️ Hybrid Intelligence

LLMs excel at language. Python excels at logic. Combining them intelligently is the engineering challenge.

🚀 Rapid Deployment

From concept to live web app in weeks, using free tools: GitHub, Streamlit Cloud, Google AI Studio.
