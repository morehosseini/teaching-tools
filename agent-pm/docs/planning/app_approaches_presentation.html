Requirements

What Drives the Architecture

💬

Conversational PIR

Progressive, multi-turn elicitation — not a 15-question form. Structured inputs (dropdowns, radio, numeric). "Why this default?" explanations. Unknown-handling with conservative/high-risk options.

🛑

Human-in-the-Loop

Two mandatory approval gates: Planning Basis approval (before WBS) and Draft Schedule review (before CPM). Edit logging with timestamps, author, field, and reason.

📚

Library-Grounded

Production rates, lead times, and logic rules are NEVER invented by the LLM. Static CSV/JSON files, version-controlled. "Not in library → planner input required."

📅

Calendar-Aware CPM

Working calendars (5/5.5/6-day), VIC public holidays, RDO rosters, Christmas shutdown, weather days. Forward/backward pass with float calculation.

✅

Validation Engine

Construction-logic rules, DCMA-style schedule quality, procurement integrity, weather/seasonal, crew continuity. Error/Warning/Info severity model.

📊

Excel Output

8-tab professional workbook: Executive Summary, PIR, Basis of Schedule, WBS, Full Schedule, Procurement, Risk Register, Validation + Sign-Off.

Option A

n8n + Web Frontend

Low-code workflow orchestration with a simple HTML form. Matches the plan's §21.2 recommendation.

Strengths

  • Visual workflow editor — fast to prototype
  • Directly matches §21.2 of the plan
  • Self-hosted or cloud deployment
  • Non-developers can understand the flow

Weaknesses

  • n8n is JS-native — complex Python logic (CPM, Pydantic, openpyxl) needs an external service anyway
  • Multi-turn conversational PIR is awkward in linear workflow model
  • Human approval gates are basic (email wait nodes)
  • State management for conversations is limited

💡 Good for a quick proof-of-concept, but the conversational PIR and rich approval UX will outgrow n8n. You'd end up building a custom frontend anyway.

3 / 7
Option B

FastAPI + Streamlit (Full-Stack Python)

Monolithic Python app. All recommended libraries run natively. Chat interface for PIR.

Strengths

  • ALL recommended libraries run natively (Pydantic, pandas, NetworkX, openpyxl)
  • Streamlit chat interface = natural conversational PIR
  • Single language, single deployment
  • Fastest path to working Phase 1a
  • Easy deploy: Docker → Fly.io / Railway
  • Ideal for Phase 1c calibration with planners

Weaknesses

  • Streamlit UI is functional but not premium
  • Session-state model can be tricky for multi-step flows
  • Scaling beyond single user requires more infrastructure
  • May need frontend rebuild for public-facing product

⭐ Best fit for Phase 1a–1c. Gets a working tool in planners' hands fastest. Core engine stays framework-agnostic for future migration.

4 / 7
Option C

Next.js + Python Microservices

Production-grade decoupled architecture. Polished React frontend, Python computation backend.

Strengths

  • Premium, fully customizable UI
  • Best for multi-user, team workflows
  • Clean separation: LLM vs computation vs presentation
  • Scales to a commercial product
  • Real-time updates via WebSockets

Weaknesses

  • 2–3× the development timeline
  • Requires JS + Python expertise
  • Over-engineered for Phase 1 validation
  • Two language runtimes to deploy and maintain

🚀 The right architecture for Phase 2+ after calibration confirms product-market fit. Premature for Phase 1.

Recommendation

Start B → Migrate to C

Build Phase 1 with Streamlit to validate the planning logic. Migrate to Next.js after calibration proves the tool's value.

Weeks 1–2: Foundation

Pydantic schemas, static CSV/JSON libraries, calendar engine (VIC_5DAY_STANDARD_2026), LLM service layer, Streamlit PIR page

Weeks 3–4: Core Logic

WBS + activity generation, procurement chains, Planning Basis approval gate, validation engine (10–12 rules), draft schedule review page

Weeks 5–6: CPM + Output

Calendar-aware CPM engine, three-point durations → P50/P80, 8-tab Excel workbook, Basis of Schedule narrative, Planner Sign-Off

Weeks 7–10: Polish

Float calculation, critical path narrative, expanded DCMA-style validation, Lock-on-Baseline, risk register with activity linkage

Weeks 11–14: Calibrate

Run against 5–8 historical projects. Measure edit rate, duration accuracy, planner satisfaction. Iterate libraries and rules.

Post-calibration: Migrate

If validated → extract core/ to FastAPI, build Next.js frontend, add PostgreSQL, authentication, team access.
